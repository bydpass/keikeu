# keikeu Road v0.1 Issue 复盘

日期：2026-07-10  
复盘基线：`main` @ `4b33bc4`  
范围：Road v0.1 的需求缺口、开发期回归、规格冲突与流程问题。  
证据：归档审计与规格、`6c2833e..9d033db` 提交链、当前测试套件和最终人工验收。

## 结论

Road v0.1 解决了开工前确认的七类产品问题：大纲格式偏差、缺少 Markdown 导出、无软删除、Archive 与 status 重合、关系录入繁琐、Library 缺少系统打开/定位、界面仍像 Material 示例。

开发过程中又暴露出五类问题：Flet API 契约理解错误、对话框无尺寸边界、关系入口语义偏离、缺失关系目标仍可交互、导出取消语义在规格内自相矛盾。这些问题均已修复并有回归测试，但它们共同说明：**纯 headless 测试不足以替代真实 Flet 目验，规格中的一句错误实现提示也会直接变成代码缺陷。**

当前没有未解决的 Road v0.1 功能阻断项。需要带入 v0.2 的主要是流程改进，而不是继续补 v0.1 功能。

## B. 开发期缺陷与规格问题

### E-01 — FilePicker 被当成 Control

- **症状**：手工启动时出现 `Unknown control: FilePicker`。
- **根因**：Planbook 直接写了挂到 `page.overlay`，但 Flet 0.85.3 的 `FilePicker` 是 `Service`；同时 `save_file()` 是 async。
- **为何漏过**：同步测试替身没有模拟真实继承关系和协程契约。
- **修复**：挂到 `page.services`，导出处理器改为 `async/await`。
- **预防**：首次使用 Flet primitive 时，先检查其 Control/Service 类型和函数签名；测试替身必须保留 async 形状。
- **状态**：已解决，Phase 2 回归测试覆盖。

### E-02 — Relation picker 展开成大块白色区域

- **症状**：关系对话框内容无边界扩张，主要操作区被挤压。
- **根因**：对话框内容和资产列表均未设置稳定尺寸与独立滚动边界。
- **为何漏过**：最初测试只验证“能添加关系”，没有验证布局约束。
- **修复**：内容宽度固定为 520，资产列表高度固定为 220，仅列表滚动；`1e0e9f3`。
- **预防**：所有 Dialog/List 组合必须有明确的尺寸与滚动所有权，并增加结构回归测试。
- **状态**：已解决，`test_relation_picker_dialog_keeps_asset_list_bounded` 覆盖。

### E-03 — 关系入口看起来像内容预览

- **症状**：关系行接近缩略卡片/原文预览，而不是跳转入口。
- **根因**：规格只定义了 picker 和展示字段，没有明确关系 UI 的主语义是“导航”。
- **为何漏过**：数据结构正确，但交互意图没有写成验收条件。
- **修复**：已有目标显示紧凑标题并可跳转；不展示原文；`d6aec93`。
- **预防**：交互规格同时写“用户要做什么”和“明确不出现什么”。
- **状态**：已解决，导航与无原文预览均有测试。

### E-04 — 丢失的关系目标仍表现为可点击

- **症状**：目标文件已删除或不在索引时，关系行仍可能暗示可跳转。
- **根因**：正常目标路径得到覆盖，缺失目标的降级状态定义较晚。
- **修复**：缺失目标只显示路径文本，不绑定点击；`64b37e5`。
- **预防**：所有文件关系功能都必须验收“存在、移动/丢失、删除”三种状态。
- **状态**：已解决，`test_missing_relation_target_row_is_not_clickable` 覆盖。

### E-05 — 导出取消语义自相矛盾

- **症状**：规格同时要求“未保存草稿先保存”以及“取消导出不改变 vault”，两条无法同时成立。
- **根因**：动作顺序没有按可撤销边界建模。
- **修复**：先显示保存对话框；用户确认目标后才持久化草稿并复制；取消时 vault 不变；`9d033db`。
- **预防**：涉及系统对话框的规格必须写清确认前、确认后、取消三个状态及其副作用。
- **状态**：已解决，`test_export_cancel_leaves_unsaved_outline_unwritten` 覆盖。

## C. 流程复盘

### 做对了什么

- 以 phase 分支和小提交推进，问题能定位到具体阶段。
- Markdown 序列化始终留在 core，系统打开和 Flet 服务始终留在 app 层。
- 用户内容保持本地 Markdown，删除采用可恢复移动，没有引入数据库或新依赖。
- 每个修复最终都补了针对性回归测试，最终全套为 112 tests。

### 哪里需要改变

1. **Issue 不应只存在于聊天、提交信息和临时审计中。** 当前 GitHub 没有 Issue，复盘需要从多个来源反向重建。
2. **UI phase 的完成门槛必须包含真实应用目验。** Headless 测试适合行为，不足以证明 Flet 控件树与布局正确。
3. **Planbook 不应写未经验证的框架接线细节。** 计划可指定行为边界，API 用法应在实现时以当前库版本验证。
4. **规格需要副作用表。** 保存、取消、删除、归档、系统打开等动作都应明确成功、取消和失败后的本地状态。
5. **产品语义要包含反例。** 例如关系入口明确是导航，不是原文预览；这比只列组件更能防止方向偏移。

## F. Pre-alpha 单用户反馈：v0.2 待处理 Issue

以下问题由当前唯一 pre-alpha 用户在 Road v0.1 验收后提出。它们是第一手使用证据，优先级高于外部竞品功能清单。

### O-01 — Finder 删除后 Library 保留幽灵条目

- **优先级**：P0
- **现状证据**：Library 的“刷新”只调用 `list_caches` / `list_outlines`；只要 `keikeu_index.json` 结构仍合法，`load_index` 就不会重新扫描 Markdown。因此在 Finder 删除文件后，旧索引条目仍会保留，打开、定位和 UI 删除都会指向不存在的路径。
- **期望行为**：用户点击刷新后，以磁盘 Markdown 为准重建索引；已被外部删除的条目从 Library 消失，并给出简短说明。
- **最小验收**：外部删除 cache 或 outline → 点击刷新 → 不崩溃、幽灵条目消失、其他条目仍可使用。

### O-02 — 单个损坏 Markdown 会中断整个索引重建

- **优先级**：P0
- **现状证据**：`rebuild_index` 使用列表推导直接调用 `read_cache` / `read_outline`，没有逐文件错误隔离；一个无法解析的 Markdown 会使整个重建失败。
- **期望行为**：健康文件照常进入索引；损坏文件以“文件损坏”条目留在 Library，并至少提供系统打开、Finder 定位和移入回收站，不让用户陷入只能手动寻找文件的死路。
- **最小验收**：一个损坏文件不影响其他资产显示；损坏条目不能进入普通编辑器，但可定位、外部打开或软删除。
- **反目标**：不自动改写、修复或覆盖用户的损坏原文。

### O-03 — 自定义结局输入框始终显示

- **优先级**：P1
- **现状证据**：`custom_ending_field` 与“结局类型”下拉框始终同时放在页面中，没有根据 `EndingType.CUSTOM` 控制可见性。
- **期望行为**：只有选择“自定义”时才显示自定义结局输入框；HE、BE、OE 时隐藏。
- **最小验收**：新建、切换选项、加载已有自定义结局三条路径显示正确；隐藏不应误写自定义正文。

### O-04 — 原作/AU/IF/PA 仍是自由文本

- **优先级**：P1
- **现状证据**：当前 `warning_setting_field` 是普通单行输入框。
- **期望行为**：改为原作、AU、IF、PA 的单选选择条；仅选择 PA 时显示“什么 PARO？”输入框。
- **最小实现建议**：保持现有 Markdown 字段不变；原作/AU/IF 直接写入 `warning_setting`，PA 写为可逆的 `PA：<用户输入>`，避免为纯 UI 改动立即引入 schema migration。
- **待定规则**：加载历史自由文本值时如何映射，需要在 v0.2 spec 中明确兼容策略。

### O-05 — 回收站是不可见黑箱

- **优先级**：P0
- **现状证据**：core 会把文件移入 `.trash/cache` 或 `.trash/outlines`，但 UI 不枚举这些目录，也没有恢复入口。
- **期望行为**：Library 提供“回收站”视图，列出软删除资产，并支持恢复和 Finder 定位。
- **最小验收**：软删除资产立即出现在回收站；恢复后重新进入正常 Library；目标文件名冲突时绝不覆盖现有文件。
- **第一版反目标**：不急于提供“清空回收站”或批量永久删除，避免新增不可逆路径。

### 建议处理顺序

1. O-01 + O-02：先恢复 Library 与磁盘事实的一致性。
2. O-05：让软删除真正可见、可恢复。
3. O-03 + O-04：集中完成配方票表单的渐进披露。

数据安全类 Issue 优先于表单便利性，不使用简单 ICE 分数覆盖这一约束。
