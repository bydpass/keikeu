# keikeu Road v0.2 — Phase 8 检查 SOP

> 用途：在作者有精力时，完成 Phase 8 的真实创作工作流检查。
> 结果填写处：[`phase8_acceptance_2026-07-15.md`](phase8_acceptance_2026-07-15.md)
> 原则：检查体验，不检查正文；可以分两天完成；任何时刻都可以暂停。

## 0. 先读这三句话

1. 这不是考试。只要把真实体验如实记下，就完成了检查的一部分。
2. 不要粘贴正文、灵感、角色、关系或 Vault 路径；只记录操作是否顺畅和问题步骤。
3. 若已经很累，今天只做“场景 A”；场景 B 留到下次。不要为了验收强行写作。

## 1. 范围与边界

本 SOP 只验证 macOS 上的真实作者工作流：

```text
Paper 准备 → Flashcard 浏览 → 返回 Paper → 系统外部 Markdown 编辑器
```

本 SOP **不**要求下列事情：

- 不要求写新正文、不要求完成一篇作品。
- 不要求迁移 Vault、删除 Paper、重命名 Paper 或测试回收站。
- 不要求验证 iCloud 远端上传、跨设备同步或网络。
- 不要求运行 pytest、编译或查看技术日志。
- 不要求找外部测试者；作者自己的真实使用即是一手证据。

若今天只想做最小检查，选一个已经存在且已保存的真实 Paper 即可。它有 Highlight 最好；没有 Highlight 也可以，跳过相关步骤并记为“无 Highlight，未测”。

## 2. 准备（约 2 分钟）

- [ ] 选择一个真实、已保存的 Paper；不要在本次检查中改动其代号或文件位置。
- [ ] 确认它的 Summary 不是空白。
- [ ] 若有 Highlight，知道至少存在一条即可；不需要阅读或记录其内容。
- [ ] 打开 keikeu，进入自己的 Vault。
- [ ] 准备在结束时填写本文件末尾的脱敏回传模板。

如果 app 无法打开、Vault 无法选择，或出现任何疑似数据异常，直接跳到“问题分级与停止规则”。不要为了继续检查反复尝试写入或迁移。

## 3. 场景 A：真实 one-shot 路径（约 5–10 分钟）

目标：确认一次真实准备 → 浏览 → 回到正文工具的闭环是否自然。

### 操作

1. 在“本地文件库”找到刚才选的 Paper，打开它。
2. 确认这是你预期的 Paper；不需要检查或复制任何正文内容。
3. 使用 Paper 页面中的“打开 Flashcard”进入卡片。
4. 从 Summary 卡开始，凭正常使用习惯浏览。
5. 若存在 Highlight，至少前进到一张 Highlight；若没有，记下“无 Highlight”。
6. 关闭或离开 Flashcard，再从同一 Paper 重新进入 Flashcard。
7. 判断当前位置是否符合你的预期；不要为了“测完”而多次来回。
8. 使用 Flashcard 的“返回 Paper”。
9. 需要继续正文时，用 Paper / 本地文件库提供的系统默认程序打开 Markdown，确认交接是否清楚。

### 当场只需回答

| 检查项 | 填写方式 |
|---|---|
| Summary-first 是否自然 | 自然 / 不自然 + 一句原因 |
| 是否频繁需要回到 Paper | 否 / 是 + 一句原因 |
| 从 Flashcard 返回 Paper | 通过 / 未通过 + 步骤 |
| 外部 Markdown 编辑器衔接 | 清楚 / 不清楚 + 一句原因 |
| 重新打开 Flashcard 的位置 | 符合预期 / 不符合预期 / 未测 |

**完成标志：** 以上五项有答案即可。没有 Highlight 不算失败。

## 4. 暂停点

场景 A 完成后可以直接停止。请在回传模板里写：

```text
本次只完成场景 A；场景 B 改日继续。
```

这不是“验收失败”，而是有意保留的分段流程。不要因疲劳把真实使用体验变成机械打勾。

## 5. 场景 B：短/中篇的跨会话重复流程（两次短会话）

目标：验证 Flashcard 位置在真实使用节奏中是否可靠，以及 Paper / Flashcard 的往返是否干扰准备流程。

### 会话 1（约 5 分钟）

1. 选择一个准备短篇或中篇的真实 Paper；可以沿用场景 A 的 Paper，也可以换一个。
2. 由“本地文件库”或 Paper 页面进入 Flashcard。
3. 正常浏览到一个**不是第一张**的位置；不用记录卡片内容。
4. 正常关闭 app 或结束本次使用。
5. 只记录：离开时是否知道自己大致停在哪里，以及过程中是否频繁想回 Paper。

### 会话 2（改日，约 5 分钟）

1. 重新打开 keikeu 和同一 Vault。
2. 从“本地文件库”找到同一 Paper，使用“打开 Flashcard”进入。
3. 判断是否回到预期的卡片位置；不要求精确背出卡片文字。
4. 在需要补充准备信息时，返回 Paper；再决定是否自然地回到 Flashcard。
5. 记录是否出现“高频、明显打断准备工作”的问题。

### 关于侧栏 Flashcard

当前已知：直接点侧栏的 Flashcard 目的地不会保留刚才的 Paper 上下文，可能要求重新选择 Paper。为避免把已知 P2 导航候选误判为位置丢失，本 SOP 的跨会话进入路径固定为：

```text
本地文件库 / Paper 页面 → 打开 Flashcard
```

若你觉得这个限制本身非常打断工作流，照实写下；它会留在下一 Road 候选池，除非实际影响达到 P1。

### 完成标志

| 检查项 | 填写方式 |
|---|---|
| 第二次进入是否回到预期位置 | 是 / 否 / 未测 |
| Paper ↔ Flashcard 往返 | 自然 / 不自然 + 一句原因 |
| 是否频繁打断准备流程 | 否 / 是 + 一句原因 |
| 是否有无法继续的问题 | 否 / 是 + 步骤 |

## 6. 问题分级与停止规则

| 等级 | 何时使用 | 处理方式 |
|---|---|---|
| **P0** | 正文或 Paper 数据丢失、被静默改写；无法安全继续；疑似迁移/删除误伤真实资产。 | 立刻停止写入和重复操作。用 Finder 复制整个 Vault 到安全位置（不覆盖原 Vault），只记录发生前的操作路径和时间；不要粘贴内容。之后再请求技术排查。 |
| **P1** | 常用主路径无法完成，或高频且明显阻塞准备工作，例如 Paper → Flashcard → 返回 Paper 反复无法走通。 | 停止继续“测试式”操作；记录最短复现步骤、出现次数、是否能绕开。无需修改正文。P1 需要修复并复验才可归档。 |
| **P2** | 能继续使用但导航、理解或效率不好，例如需要多一步重新找到 Paper。 | 记录一句体验和最短步骤，放入下一 Road 候选池；不扩大当前 Phase。 |
| **P3** | 纯偏好、措辞、视觉或锦上添花想法。 | 可选记录；不影响 Phase 8 结论。 |

### 永远优先保护数据

出现疑似 P0 时，不要：

- 重启迁移；
- 尝试“修复” Markdown；
- 删除、移动或覆盖原 Vault；
- 把真实 Paper 发到对话里。

只需保留原状、复制 Vault 作保护、写下脱敏步骤即可。
