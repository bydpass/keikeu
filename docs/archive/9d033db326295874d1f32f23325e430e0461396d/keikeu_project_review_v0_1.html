1. 项目主发心

keikeu 的主发心只有一个。

当创作者记录灵感时，指引记录者将其变为一份大纲。

项目 slogan：

存住一瞬的灵光

这个发心必须保持单核。keikeu 不做大而全创作平台，不做 AI 写作器，不做同人社区，不做约稿交易工具。

2. 目标用户

keikeu 的目标用户是且仅是：

有能力动笔写作的同人创作者，尤其是单机同人女。

目标用户特征

  • 会产生大量灵感、口嗨、场面执念。
  • 能写作，但常卡在“灵感 → 大纲 / 正式动笔”。
  • 重视隐私。
  • 可能排斥 AI 直接介入创作。
  • 不需要社交平台。
  • 更需要本地、轻量、可迁移的创作工具。

第一版不服务

  • 约稿平台
  • 社区运营
  • 读者互动
  • 多人协作
  • 云端同步
  • AI 代写

3. 产品定位

keikeu MVP 的定位：

本地优先 Markdown 大纲生成器 + 灵感 cache。

核心链路：

乱灵感 / 口嗨
  ↓
灵感 cache
  ↓
配方票填写 / 生成
  ↓
本地 Markdown 大纲

一句话：keikeu 把同人创作者的灵感从“口嗨火星子”整理成“可继续写作的大纲文件”。

4. 产品立场

创作者资产属于创作者，不属于软件。

第一版必须满足：

  • 本地优先
  • 无账号
  • 无云同步
  • 无社区
  • 无外部 fandom / 角色 / CP 数据库
  • 无 AI 强依赖
  • 用户自行选择存储位置
  • cache 与 outline 都保存为 Markdown
  • index.json 只作为辅助索引，不作为资产本体

如果 keikeu 停止维护，用户仍然应该能打开自己的 .md 文件继续写。

“大纲”定义为一份 Markdown 文件。

保留最初的灵感、口嗨、碎片文本。

## 2. 整理后摘要

这篇大概讲什么。

## 3. Fandom + 人物 / CP

- Fandom：
- 人物：
- CP：

## 4. 观前提醒

- AU / IF / PA：
- CP 结构：
- 情节元素：

## 5. 流水账

故事大概怎么走。

## 6. Ending Type

HE / BE / OE / 自定义

## 7. 与其他灵感的逻辑关联（Optional）

- 关系：
- 关联对象：
- 说明：

关键决策

  • Rough Summary 拆成“原始灵感”和“整理后摘要”。
  • 原始灵感必须保留，不能被整理后摘要覆盖。
  • 流水账默认自由文本，可选模板。
  • Ending Type 固定枚举只保留 HE / BE / OE + 自定义。
  • 逻辑关系只能连接本地已有 cache / outline。
  • 逻辑关系枚举必须少。

6. 灵感 cache

cache 不是完整备忘录系统。

cache 是灵感的临时收容层，等待被炼成大纲。

cache 的职责是收住火星子，不替代 Obsidian、Notion、Apple Notes。

最小字段

  • 标题
  • 原始灵感
  • 临时备注
  • 创建时间
  • 更新时间
  • 状态
  • 关联 outline 路径

状态

raw刚存，未处理
drafting已开始转配方票
outlined已生成大纲
archived封存

cache 也保存成 Markdown。index.json 只负责辅助读取、搜索、筛选和重建状态。

7. 本地存储结构

用户首次启动时选择本地文件夹作为 keikeu vault。

keikeu-vault/
  cache/
    2026-06-19-2130-黄光床边灵感.md
  outlines/
    2026-06-19-阿德琳等着姐姐.md
  .trash/
    cache/
    outlines/
  keikeu_index.json

Markdown 是资产本体。JSON 是软件索引。

索引坏了，可以从 Markdown 重建。Markdown 丢了，索引没有意义。

8. MVP 页面结构

8.1 灵感 Cache 页面

  • 新建灵感
  • 编辑灵感
  • 保存灵感
  • 转成大纲
  • 封存
  • 删除

禁止复杂标签、社交分享、推荐流、红点催办、待处理数量焦虑。

8.2 配方票 Editor 页面

  • 编辑大纲字段
  • 从 cache 填入原始灵感
  • 保存 outline
  • 导出 Markdown
  • 添加逻辑关联
  • 检查缺失字段

字段允许留白。软件可以温和建议补完，但不能强制。

8.3 本地文件库

轻量文件列表，不做完整项目管理系统。

  • 查看 cache
  • 查看 outlines
  • 搜索标题
  • 按状态筛选
  • 打开文件
  • 在系统文件夹中显示

9. 当前 0 号用户反馈

当前 UI 已能运行，体验未达 MVP 标准。

  1. 画面过于“安卓”，不符合开发者本人，也不符合目标用户审美。
  2. 不能快捷删除 note 和 outline。
  3. 添加逻辑关系的操作过于繁琐且不切实际。
  4. outline 格式与预想存在偏差，需要修正。
  5. status 的归档操作和 archive 按钮功能重合。
  6. 暂不能导出 Markdown 文件。

当前版本只是 Flet demo 长出了 keikeu 器官，还没有真正长成 keikeu。

10. Road v0.1

Phase 0：保存当前可运行状态

  • 确认当前 UI 能跑。
  • 确认 warning 已消失。
  • commit 当前绿色状态。
git add .
git commit -m "chore: stabilize initial flet app shell"

Phase 1：修 MVP 脊椎：Markdown 输出

  • 修正 outline schema。
  • 保存 outline 时生成正确 Markdown 文件。
  • 空字段允许留白或写“待补”。
  • 文件必须能被普通文本编辑器打开。
git commit -m "feat: write outline markdown with v0 schema"

Phase 2：实现导出 Markdown

  • 保存：写入 keikeu vault 内部 outlines/。
  • 导出：让用户选择目标位置，复制或另存当前 .md。
  • 第一版只支持 Markdown，不做 PDF、DOCX、HTML、图片。
git commit -m "feat: export outline as markdown"

Phase 3：删除与归档重构

  • 保留 Archive / 封存按钮。
  • 不让用户手动在 status 下拉中选择 archived。
  • Archive 按钮负责把状态改为 archived。
  • 删除采用 soft delete。
删除 note / outline
  ↓
文件移动到 .trash/
  ↓
index 更新
  ↓
列表中隐藏

理由：创作者资产不能被手滑处决。

git commit -m "fix: simplify archive and add soft delete"

Phase 4：重做逻辑关系 picker

添加链接
  ↓
界面内选择目标笔记 / 大纲
  ↓
选择关系
  ↓
关系建立

关系枚举：前作、续作、IF、外传、同系列。

禁止手动输入文件路径、手写文件名、复杂知识图谱、复杂节点网络。

git commit -m "feat: add local relation picker"

Phase 5：视觉方向修正

当前视觉问题：太安卓，太像 Flet demo，不像 keikeu。

目标气质本地小册子、轻量写作桌、同人作者的灵感票夹、paper / notebook / zine / local archive
避免气质安卓设置页、企业后台、蓝色效率工具、Material demo、钉钉味
  • 降低 Material icon 存在感。
  • 减少塑料感圆角和阴影。
  • 使用更轻的 sidebar。
  • 增加留白。
  • 文案换成创作者心智。

推荐文案：新灵感、炼成大纲、封存、先放着、导出 Markdown、检查缺什么。

git commit -m "style: soften app shell visual language"

11. Post-MVP

灵光回潮

用户再次打开软件时，温和浮现旧灵感。

  • 不进 MVP。
  • 不做红点。
  • 不做催办。
  • 每次最多浮现 1 条。
  • 只提供：炼成大纲 / 先放着 / 封存。

写作调理

帮助用户从大纲进入写作准备。

  • 检查大纲缺口。
  • 提醒冲突不足。
  • 整理人物动机。
  • 协助拆第一段。
  • 协助选择开头方式。

原则：不代写正文，不制造 KPI，不催用户完成。

12. 当前最高优先级

现在不要同时修六个问题。

下一步只做：P0-001 修正 outline markdown schema。

理由：文件格式是 keikeu 的骨架。骨架不正，UI 再漂亮也是尸体化妆。

  1. 导出 Markdown

13. 当前项目总判断

keikeu 已经完成最危险的一步：从“AI 创作工具幻想”收缩成“本地创作者资产工具”。

现在的核心风险不在技术，而在范围失控。

必须警惕

  • 又开始做大入口。
  • 又开始做复杂图谱。
  • 又开始接 AI。
  • 又开始搞社交。
  • 又开始做漂亮 UI 却忘记 Markdown 输出。
  • 又开始把创作者资产锁进软件黑箱。

keikeu 第一版必须像一把小刀：收住灵感，切出大纲，把文件交还给创作者。

MVP 的唯一验收：

用户输入一段混乱口嗨，能在 3 分钟内得到一份可保存、可编辑、可继续写作的本地 Markdown 大纲。

HTML v0.1 - 单文件、无外部依赖、暖纸面视觉、caveman readable。
