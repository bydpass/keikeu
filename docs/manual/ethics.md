# keikeu 技术伦理手册

keikeu 处理的是作者尚未公开、尚未定稿的创作材料。这里的核心问题不是“技术能不能做”，而是：**这项能力会把控制权、收益与风险交给谁？**

## 1. 四个先问问题

任何新功能先回答四题：

1. **授权：** 输入属于谁？当事人是否知道、同意并能撤回？
2. **权力：** 功能增强谁的控制，又削弱谁的控制？
3. **风险：** 最坏的合理用法是什么，代价由谁承担？
4. **退出：** 用户能否拒绝、删除、导出，并用普通工具修复数据？

只要其中一题没有可信答案，就不该默认启用。若答案改变产品边界，应先修改 `SPEC`，而不是让实现偷偷先行。

## 2. 五个观察角度

| 角度 | 它追问什么 | 对 keikeu 的提醒 |
| --- | --- | --- |
| 后果 | 总收益是否掩盖了集中在少数人身上的伤害？ | 便利不能由作者失去控制来支付。 |
| 原则 | 即使结果看起来不错，是否仍越过了不可交换的边界？ | 可见不等于获准；可抓取不等于可使用。 |
| 关系 | 谁更脆弱，谁更依赖语境和匿名？ | 私密草稿、圈层语境与身份线索需要更小的数据面。 |
| 正义 | 收益、风险和纠错成本是否由同一方承担？ | 工具或平台不能获益，却把暴露与追责成本留给创作者。 |
| 品格 | 产品反复鼓励用户成为什么样的人？ | keikeu 应帮助作者整理自己的灵感，而不是消费、比较或复制他人作品。 |

这些角度不会自动给出答案；它们用来暴露“更方便”背后的交换。

## 3. 与 keikeu 直接相关的风险

### 语境坍塌

内容从原来的私人或小圈层语境进入更公开、可检索、可聚合的环境。内容本身没变，暴露面和风险已经变了。

### 外部性

使用者或产品获得效率，把误用、暴露、纠错和维权成本转给素材的原作者。

### 功能漂移

一项窄功能逐步扩张：手动记录自己的灵感，变成链接导入、自动抓取、公共聚合、推荐与排名。每一步都看似很小，终点却已是另一种产品。

### 平台化与监控

账号、社交关系、榜单、推荐、行为分析和内容画像会把私人创作工具变成影响用户行为的平台，也创造原本不需要存在的敏感数据。

### 自动化权威

总结、分类或模型输出如果不可解释、不可编辑，会让系统的判断替代作者判断。输出越像“正确答案”，作者越难保留最终决定权。

## 4. 评审一项功能

按这个顺序评审，答案写短句即可：

1. **目的：** 它是否直接加强 `已有灵感 → Paper → Flashcard → 外部编辑器`？
2. **输入：** 只处理用户有权提供的内容吗？
3. **数据流：** 内容写到哪里、是否离开设备、谁能读取？
4. **默认值：** 最保护作者的行为是否无需额外设置？
5. **失效方式：** 出错、离线或第三方不可用时，会不会损坏或困住作者资产？
6. **滥用方式：** 恶意用户能否借它抓取、仿写、排名、曝光或画像他人？
7. **退出方式：** 用户能否拒绝、撤回、导出和手工修复？
8. **归属：** 最终决定应写进 `SPEC`、`RULES`、ADR，还是根本不需要存在？

这个清单是讨论工具，不是新的验收标准。产品验收仍由 `SPEC` 定义。

## 5. 当前边界的例子

| 提议 | 伦理判断 | 当前 truth |
| --- | --- | --- |
| 保存用户自己输入的 Summary 与 Highlights | 输入权属清楚，Markdown 可检查、导出和修复 | [`SPEC §3–4`](../SPEC.md#3-author-asset-contract) |
| 将用户选择的文件服务目录当普通路径 | keikeu 不接管账号、同步时机或冲突合并 | [`SPEC §7`](../SPEC.md#7-vault-and-network-boundary) |
| 自动抓取外部作品或建立作者数据库 | 授权不清，扩大语境并把风险转给原作者 | [`SPEC §11`](../SPEC.md#11-explicit-non-goals-before-acceptance) |
| 默认上传草稿或记录内容行为 | 制造不必要的数据与监控面 | [`RULES §3`](../RULES.md#3-author-text-and-privacy) |
| 让生成结果不可编辑 | 系统判断压过作者判断 | [`SPEC §3`](../SPEC.md#3-author-asset-contract) |
| 公共评分、推荐、榜单或社区 | 改变权力结构，也把工具推向平台 | [`SPEC §11`](../SPEC.md#11-explicit-non-goals-before-acceptance) |

例子说明“为什么”，不冻结未来。若开发者决定改变边界，应先在唯一 truth 中明确范围、数据流、默认值和退出方式。

## 6. 常用术语

| 术语 | 最短定义 |
| --- | --- |
| Consent | 对明确用途的知情、可拒绝、可撤回授权。 |
| Opt-in | 默认不启用，由用户主动开启。 |
| Data minimization | 只处理完成当前任务必需的数据。 |
| Privacy by design | 从数据流和默认值开始保护隐私，而不是事后补声明。 |
| Context collapse | 内容脱离原语境后获得意外的受众、可见性或用途。 |
| Contestability | 用户能看懂、反驳、修改或撤销系统判断。 |
| Dual use | 同一能力既能支持正当目标，也容易被用于伤害。 |
| Function creep | 数据或能力逐步被用于最初目的之外。 |

## 7. 最短结论

keikeu 的伦理方向可以压成一句话：

> 帮作者组织自己的创作材料，同时让内容、决策和退出权继续留在作者手里。

需要判断“现在允许做什么”时，离开本页，回到 [`SPEC`](../SPEC.md) 与 [`RULES`](../RULES.md)。
