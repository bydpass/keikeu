01 · executive decision

三个不同层级的状态，必须分开说

已完成 工程功能

Phase 0–7 已按独立分支与提交完成：Paper v2、迁移、索引恢复、UI、Flashcard、Outline 退休和 macOS 文件服务 smoke 都有实现与记录。

进行中 产品体验验收

Phase 8 已建立脱敏验收模板并获得部分反馈，但真实 one-shot 与短/中篇跨会话流程尚未完整回传。

尚未发生 Road 归档 / Tag

不能仅凭 78 个自动化测试或合成 Vault smoke 宣称 macOS MVP 已验收；是否打 tag 仍由开发者决定。

!

当前阻塞不是代码故障，而是缺少真实使用证据

剩余验收需要作者在不暴露正文的前提下，确认 Summary-first 是否自然、是否频繁离开 Flashcard，以及短/中篇的跨会话位置和 Paper/Flashcard 往返。已记录的“侧栏 Flashcard 未保留当前 Paper 上下文”按作者分级为 P2 候选，不是 Phase 8 要修的 P0/P1 阻塞。

02 · old → new architecture

从 Cache / Outline 双主角，收敛成 Paper 为唯一资产

目录名 cache/ 为了兼容 Vault 路径而保留；其中的内容已不再是 v0.1 Cache，而是带 type: paper 与 schema_version: 2 的 Paper Markdown。

v0.1 · 双 schema 工作流

创作状态、Cache 与 Outline 共同构成活动路径，关系与大纲依赖也进入了日常 UI。

Vault 资产cache/*.md（Cache）＋ outlines/*.md（Outline）
↓
Core 投影Index 同时维护 caches / outlines；CacheStatus、Relation 参与模型。
↓
Flet 主流程Cache 编辑器 → Outline 编辑器 / relation picker；没有 Flashcard。
↓
风险双资产协议、迁移删除顺序与旧 UI 入口容易互相牵连。

v0.2 · Paper–Flashcard Core

Markdown Paper 是唯一活动创作资产；JSON 与设备状态都可丢弃、可重建。

Vault 资产cache/*.md（Paper v2）＋ .trash/cache/ ＋ 可重建 keikeu_index.json
↓
纯 Python Core（零 Flet import）Paper 模型 / Markdown I/O / Index rebuild / 回收站恢复 / 一次性 v0.1 迁移器。
↓
Flet AppPaper 编辑、Summary-first Flashcard、本地文件库、v0.1 迁移确认门。
↓
设备外状态~/.keikeu_state.json 只保存卡片位置；正文仍用系统外部编辑器。
维度v0.1v0.2 当前实现意义
主资产Cache + OutlinePaper v2 Markdown单一可手工修复的作者资产协议。
创作内容标题、原始灵感、临时备注、状态代号、冻结初稿副本、可编辑 Summary、Highlights、Tags把准备工作与正文写作明确分开。
卡片体验没有Summary-first，只读投影；当前卡位在设备本地不制造第三份正文或“完成状态”。
索引Cache / Outline 双索引v2 papers + errors坏文件隔离，不阻断其他 Paper；可由 Markdown 重建。
删除与恢复旧双目录只处理 Paper 的 .trash/cache软删除保持字节；代号冲突必须显式改名或取消。
迁移不适用v0.1 只读预检 → Vault 外完整备份 → staging 转换 → 原子切换先保护原资产，再退休 Outline。
UI 导航Cache / Outline / LibraryPaper / Flashcard / 本地文件库Outline 与关系 picker 已从正常路径退休。
文件服务路径输入系统目录选择 + 手工路径 fallback普通目录与已挂载 iCloud Drive 均被当作本地路径。
非目标—无云端、帐号、遥测、provider API、AI 代写或内置正文编辑器保留作者控制和 local-first 边界。

04 · evidence boundary

哪些已经被验证，哪些还没有

已有证据

  • 最后一次完整自动化测试记录为 78 passed；随后 Phase 8 为文档变更。
  • 所有 Core 保持零 Flet import；扫描未见第三方网络请求、帐号或后台同步实现。
  • 以用户提供的、已备份的 v0.1 Vault 副本演练迁移：2 个 Cache 转为 Paper，Outline 从活动 Vault 移除，外置备份和迁移报告存在。
  • 在合成的普通目录和 iCloud Drive Vault 完成可见 Flet smoke：目录选择、Paper、Flashcard、恢复和冲突隔离。
  • 作者简测已确认：可从 Flashcard 返回 Paper、可由系统默认程序打开 Markdown，未观察到 P0/P1。

还不能声称的事

  • 未完整确认真实 one-shot 的 Summary-first 感受与频繁离开 Flashcard 的情况。
  • 未完整确认真实短/中篇在至少两个会话中的位置恢复和来回流程。
  • iCloud smoke 不证明远端上传、跨设备时序、离线占位文件或冲突副本到达时机。
这不是“卡住不做事”的技术阻塞：代码阶段已经结束。它是刻意保留的产品证据门槛，避免把合成 smoke 当成作者真实工作流。

05 · proposed guardrails

建议补充给我的规矩

以下是本轮暴露出来、值得长期执行的协作约束。它们先记录在本状态页，尚未修改 gitagent.md；若你认可，再把它们正式纳入项目规则。

01

证据分级，不许混写

任何“通过”都标明是单元测试、合成 smoke、真实 Vault 演练还是作者真实创作。低等级证据不能替代高等级验收。

02

Phase 关门前做一次事实表

每个 Phase 结束必须列出：分支、commit、改动、实际运行命令、未验证项与资产风险；没有事实表不切下一个 Phase。

03

作者资产操作双确认

涉及真实 Vault 的迁移、覆盖、移动、删除或 GUI 文件选择，在执行当下再确认；合成资产不能默认为真实资产的代理。

04

产品结论与工程结论分离

测试绿只能证明实现满足已编码的契约；只有真实作者工作流可以支撑“好用”“MVP 已验收”“可归档”这类产品结论。

05

上下文预算提前治理

到约 60% 先总结正在做的分支、证据与未决项；超过你指定的 69% 前必须 compact，再继续改变仓库状态。

06

候选改进不偷渡进当前 Phase

P2/P3 想法进候选池；除非你重新授权或它被验证为 P0/P1，否则不在验收阶段顺手改产品行为。

07

Git 记录遵守边界

实施 Phase 必有独立分支与聚焦 commit；非 Phase 的报告、调研或维护文档要明确是否纳入当前分支，绝不模糊混入。

08

配置副作用也要报备

UI smoke 若改变默认 Vault 或设备本地状态，必须记录影响；除非你授权，不把“恢复现场”当成可以静默执行的操作。

09

提交前复核，默认不代替你提交

先给出 diff、检查结果和准确 commit 建议；只有你明确说“commit”时才 stage / commit，永远不 push 除非你再次明确要求。

06 · what this agent did

我在这一轮实际完成了什么

审计与基线
对照 spec、planbook、appdesign 与旧代码；冻结含正常、空字段、损坏 Cache 和 Outline 的 v0.1 合成迁移夹具，避免迁移从真实唯一 Vault 开始试错。
核心契约
实现 Paper v2 的模型、验证与 Markdown 原子读写：首次保存冻结初稿副本，后续更新不允许改写它；显式 rename 不覆盖同代号 Paper。
安全迁移
实现一次性 v0.1 迁移器：先只读预检，再建 Vault 外完整备份，在 staging 中转换和验证，最后原子切换；测试受控失败和恢复路径。
Vault 与恢复
把 index 收敛成可重建的 Paper 投影，隔离坏 Markdown；实现字节保持的软删除与显式冲突恢复。
新 UI 闭环
用 Paper 编辑器、本地文件库和 Summary-first Flashcard 替换旧 Cache / Outline 主交互；正文仍交由用户的系统默认外部编辑器。
状态与退休
把 Flashcard 位置放在 Vault 外的设备本地 JSON；添加 v0.1 迁移确认页，移除正常运行路径中的 Outline 页面、关系 picker 与旧 schema I/O。
macOS 文件行为
接入系统目录选择与路径 fallback；阻止外部修改、删除或移动后的静默覆盖；以合成普通目录与 iCloud Drive Vault 做可见 smoke，不读取作者正文。
真实副本演练
仅使用你明确提供的、已备份的 v0.1 Vault 副本完成迁移演练；该副本和备份目录已 gitignore，未被提交。
Phase 8 管理
建立脱敏验收模板，录入你的部分简测结果；把“侧栏没有保留当前 Paper 的 Flashcard 上下文”定位为 P2 候选，没有越过授权在验收阶段擅自改 UI。
Git 纪律
为每个实施 Phase 使用独立分支与聚焦提交；未 push。当前工作树在生成本报告前为干净状态。

生成范围：本状态页基于当前工作树、Road v0.2 规格/Planbook、审计记录与 Phase 8 部分验收记录整理；它是静态快照，不会自动追踪后续 Git 状态。

下一件正确的事：完成两类真实但脱敏的作者验收结果；若出现 P0/P1，则在新的明确授权下修复并复验；若没有，再做 Road 归档复核与开发者决定的 tag。
