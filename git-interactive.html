    边界 / 风险

    IGit 到底存什么

    Git 是一个按内容寻址的对象库，外加一组会移动的名字。它的核心对象很少：

    blob一份文件内容；不带文件名。
    tree一次目录结构：文件名、模式，以及它们指向的 blob / tree。
    commit指向一棵 tree、父提交、作者与说明；因此形成历史图。
    ref分支或标签的名字。分支会移动，提交对象本身不改。

    为何哈希值重要

    对象内容决定对象 ID。文件、目录或提交内容不同，ID 就不同；所以 Git 能复用相同内容，也能发现历史中的精确差异。

    要点：commit 不是“改了三行”的补丁包，而是一张带父指针的项目快照。

    II分支、HEAD 与合并

    分支只是指向提交的标签。HEAD通常指向当前分支。提交一次，当前分支向新提交前进一步。

    动作真实含义历史形状
    switch -c从当前提交创建新分支并切过去暂时没有新提交
    fast-forward目标分支没有分叉，只把指针向前移一条直线
    merge commit两边都有新提交，生成一个有两个父提交的新提交分叉后汇合
    conflictGit 无法自动决定同一处内容应采用哪边等待人类解决，不是数据已经丢失

    冲突处理

    git status → 逐个读冲突双方 → 手动保留正确意图 → git add <file> → 测试 → 完成合并。无法判断时用 git merge --abort 回到合并前。

    III本地与 GitHub 的边界

    命令方向是否改工作区
    git fetch远端 → 本地的远端跟踪引用否
    git pull先 fetch，再把远端变化整合进当前分支可能会
    git push本地提交 → 远端分支否，但会改变共享远端
    origin/main上次 fetch 后，你本地记住的远端 main 位置不是实时网络状态

    keikeu 的稳妥习惯

    先 git fetch 与 git status 看清分歧，再决定 merge、pull 或 push。推送会改变 GitHub，agent 必须得到明确授权。

    IV安全操作册

    先问：我要撤销哪一层？

    • 只想取消暂存、保留文件修改：git restore --staged <file>
    • 想丢弃某文件未暂存修改：先看 diff，再用 git restore <file>
    • 已推送的坏提交：用新提交 git revert <hash> 反向撤销
    • 临时收起工作：git stash push -m "说明"，但 stash 不是仓库

    不要凭感觉执行

    • git reset --hard：会一起丢弃工作区修改
    • git clean -fd：会删除未跟踪文件
    • git push --force：会改写远端历史
    • git branch -D：会强删尚未合并的分支

    紧急时先停手：git status、git diff、git log --oneline --decorate --graph --all -n 20。
