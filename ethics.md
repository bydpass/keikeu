# 0. 技术伦理学到底问什么？

核心就四个问题：

1. **我能不能做？**
   技术上可行，不等于应该做。

2. **我有没有权利做？**
   数据、内容、用户行为、社区边界，谁授权了？

3. **谁受益，谁背锅？**
   平台吃增长，用户/创作者/弱势群体承担风险，这是红灯。

4. **出事后谁能纠正？**
   有没有删除、申诉、退出、审计、追责机制？

套到 barn：
它不是“功能设计不周”，而是这四题基本都挂科。

---

## 1. 最基础伦理理论

### 1. 功利主义 / Consequentialism

看结果。
一个东西带来的总好处是否大于总伤害。

技术产品里常见说法：

> “这个功能能帮更多人，所以值得做。”

风险：它很容易牺牲少数人。
比如：读者找粮更方便，但作者失控、出圈、被开盒。结果就是**便利压死人**。

keikeu 里对应判断：

> “这个功能方便吗？”不够。还要问：“它会不会把风险转嫁给创作者？”

---

### 2. 义务论 / Deontology

看原则。
有些事即使有好结果，也不能做。

比如：

* 未经同意抓取他人作品，不行。
* 未经许可拿作品训练模型，不行。
* 把私人/半私人语境里的内容搬到公共入口，不行。

这套很适合写 constitution。

一句话：

> **有些边界不能拿效率抵押。**

---

### 3. 德性伦理 / Virtue Ethics

看你成为什么样的开发者、平台、社区成员。

它问的不是“这次有没有违法”，而是：

> 这个产品会训练出什么样的人？

评分系统会训练读者变成消费者。
推荐算法会训练用户追热度。
公开榜单会训练作者焦虑、互害、媚算法。

keikeu 应该训练用户成为什么？

> 更清楚自己想写什么的创作者。
> 不是更会消费别人作品的仓鼠。

---

### 4. 关怀伦理 / Ethics of Care

看关系、脆弱性、依赖。

这套特别适合同人圈。因为同人不是冷冰冰的内容市场，而是关系网络：作者、读者、亲友、圈层、雷点、匿名、安全感。

它会问：

> 这个功能有没有照顾到最容易受伤的人？

比如小透明作者、敏感题材作者、锁文作者、女性向圈子、匿名创作者。
技术伦理里最容易被忽略的，就是这些“不像用户画像里主流人群”的人。

---

### 5. 正义论 / Justice

看资源、权力、风险是否公平分配。

技术产品经常搞出这种结构：

```text
收益：平台、读者、大用户
成本：作者、小用户、边缘群体
```

这叫不正义。

barn 的结构大概就是：

```text
读者得到便利
APP 得到流量
作者得到失控和风险
```

经典垃圾交易。🚮

---

## 2. 技术伦理核心术语

### 1. Consent / 同意

不是“公开可见就默认同意”。
真正的同意至少要满足：

* 明确知道发生什么
* 可以拒绝
* 可以撤回
* 拒绝后不会被惩罚
* 同意范围清楚

技术产品常见烂活：

> “你发网上了，所以我可以抓。”

这是强盗逻辑披了技术外套。

---

### 2. Opt-in / Opt-out

**Opt-in：默认不加入，用户主动同意。**
**Opt-out：默认加入，用户自己退出。**

伦理强度：

```text
Opt-in > Opt-out > 没得选
```

同人领域建议默认 opt-in。
因为作者边界敏感，不能默认征用。

---

### 3. Privacy by Design / 隐私内建

不是出事后补隐私政策，而是架构上少收数据。

原则：

* 能本地就本地
* 能不收就不收
* 能不上传就不上传
* 能匿名就匿名
* 能删除就真删除

keikeu 的正确路线：

```text
local-first
no account by default
no cloud by default
no analytics by default
```

---

### 4. Data Minimization / 数据最小化

只收完成任务必需的数据。

keikeu 只需要：

```text
用户自己的灵感
用户自己的 brief
本地 Markdown
```

不需要：

```text
作者数据库
作品数据库
阅读历史
CP 热度
外部平台抓取数据
```

少收数据不是功能弱，是少造雷。💣

---

### 5. Context Collapse / 语境坍塌

内容从原本的小圈层、半私密语境，被搬到陌生大环境。

同人圈超级关键。

例子：

```text
AO3 上的文 → 被搬到国内 APP → 被评分 → 被围观 → 被举报/挂人
```

作品没变，语境变了，风险就变了。

这就是 barn 事件的核心之一。

---

### 6. Boundary / 边界

技术伦理里，边界不是情绪词，是系统设计对象。

边界包括：

* 作者边界
* 平台边界
* 圈层边界
* 数据边界
* 公开/私人边界
* 人类创作/AI 处理边界

keikeu 的边界句：

> **只处理用户自己的灵感，不处理他人作品。**

---

### 7. Accountability / 问责

出事后谁负责？

一个产品不能只说：

> “用户自己上传的，我们不负责。”

这句话法律上也许能挡一点，伦理上很烂。

问责设计包括：

* 删除机制
* 日志
* 版本记录
* 明确禁止用途
* 滥用处理
* 用户可导出/迁移
* 开发者公开原则

---

### 8. Transparency / 透明度

用户要知道系统在做什么。

特别是 AI 产品：

* 有没有调用远程模型？
* 输入会不会上传？
* 会不会被保存？
* 会不会用于训练？
* 输出是模板生成还是模型生成？
* 数据存在哪里？

keikeu 以后如果接模型，必须写清楚：

```text
默认不上传。
远程模型调用必须手动开启。
开启前显示数据流向。
```

---

### 9. Explainability / 可解释性

系统输出不能像神谕。

对 keikeu 来说，不需要复杂 AI explainability，但需要输出结构清楚：

```text
我为什么把这个识别为饺子醋？
为什么建议这个成品形态？
为什么列出这些禁忌？
```

这会让用户保留主导权。

---

### 10. Contestability / 可争辩性

用户能不能反驳系统？

AI 说错了，用户能不能改？
系统分类错了，用户能不能删？
自动标签错了，用户能不能覆盖？

keikeu 必须坚持：

```text
所有输出可编辑。
用户永远是最终作者。
```

## 3. 技术伦理常见风险词

### 1. Externality / 外部性

产品收益内部化，成本丢给别人。

barn：

```text
APP 得流量
读者得便利
作者承担出圈风险
```

这就是外部性。

keikeu 要避免：

```text
用户爽了，其他作者被暴露
```

---

### 2. Dual-use / 双重用途

一个工具既能做好事，也能作恶。

比如：

* 灵感整理器：帮助创作
* 同样的结构化能力：也能做洗稿 brief、仿写指令、搬运索引

所以你要写 abuse case table。
不要幻想用户全是天使。互联网默认哥布林模式。🧌

---

### 3. Misuse / 滥用

用户偏离产品原意使用。

例子：

```text
把他人全文贴进 keikeu，让它总结成约稿 brief
```

产品决策：

```text
MVP 不支持长文解析。
输入提示写明：请只输入你自己的灵感。
```

---

### 4. Function Creep / 功能漂移

一开始功能很干净，后来慢慢滑坡。

路径通常这样：

```text
私人灵感整理
→ 支持链接
→ 支持批量链接
→ 支持自动抓取标题
→ 支持抓取摘要
→ 支持公共分享
→ 支持推荐
→ 粮仓
```

这就是你最该防的。

---

### 5. Platformization / 平台化

工具变平台。

工具服务用户目标。
平台重塑用户行为。

keikeu 一旦有账号、feed、推荐、点赞、榜单、作者页，它就从工具变平台。
MVP 必须远离。

---

### 6. Surveillance / 监控

不只是摄像头。
产品记录用户行为、偏好、阅读、CP、雷点、XP，也可能构成监控。

同人项目尤其敏感，因为这些数据很私人。

所以：

```text
不要做用户画像。
不要做行为分析。
不要做内容 analytics。
```

---

### 7. Data Colonialism / 数据殖民

把别人的文化、劳动、表达拿来当原料，转化成平台资产。

AI 时代很常见：

```text
创作者写文
平台抓取
模型训练
公司赚钱
创作者失控
```

同人圈对 AI 敏感，不是无知，是看懂了这个结构。

---

### 8. Algorithmic Bias / 算法偏见

系统倾向性地伤害某些群体。

在同人工具里可能表现为：

* 主流 CP 被更好识别
* 小众 CP 被误判
* 女性向表达被标签化
* 性少数题材被错误归类为高风险
* 非标准中文/口嗨被模型整理成办公废话

keikeu 的抵抗方式：

```text
不做公共推荐算法。
输出可编辑。
用户自定义 fandom / CP / tag。
```

---

## 4. 产品伦理的四个硬测试

你以后每加一个功能，问这四题。

### Test 1：Consent Test

```text
这个功能处理的数据，是否得到了明确同意？
```

没同意，默认不做。

---

### Test 2：Power Test

```text
这个功能增强了谁的权力？
削弱了谁的控制？
```

如果增强平台/读者，削弱作者，红灯。

---

### Test 3：Worst-case Test

```text
最坏的人会怎么用它？
```

不要只想理想用户。
要想黑粉、搬运狗、开盒人、AI 洗稿仔。

---

### Test 4：Exit Test

```text
用户能不能退出、删除、撤回、导出？
```

没有 exit，产品就是笼子。

---

## 5. 直接套 keikeu：你的伦理底线

你项目的最小伦理系统可以这样写：

```md
# keikeu Ethical Baseline

1. keikeu is a personal creative tool, not a fandom content platform.
2. User-owned input only.
3. No scraping.
4. No reposting.
5. No public rating.
6. No ranking.
7. No public aggregation.
8. No creator database.
9. No fanwork database.
10. No unauthorized AI training, summarization, imitation, or embedding.
11. Local-first by default.
12. Remote model calls must be opt-in.
13. All generated output must be editable.
14. Users remain the final authors.
15. When convenience conflicts with creator boundary, boundary wins.
```

这就是你的技术伦理地基。够 caveman，也够硬。

## 6. 入门学习路线

按这个顺序学，不容易脑子糊：

### 第一层：伦理学底座

先懂四套：

```text
功利主义：看后果
义务论：看原则
德性伦理：看人会变成什么样
关怀伦理：看关系和脆弱者
```

### 第二层：技术产品术语

重点掌握：

```text
consent
privacy by design
data minimization
accountability
transparency
dual-use
misuse
function creep
platformization
context collapse
```

### 第三层：应用场景

拿这些问题练手：

```text
爬虫伦理
AI 训练数据
推荐算法
社交平台治理
内容审核
隐私设计
开源责任
创作者版权
社区边界
```

### 第四层：写自己的项目宪法

你的输出不该是读书笔记。
应该是：

```text
CONSTITUTION.md
DATA_POLICY.md
ABUSE_CASES.md
README ethics section
```

学习最好的方式就是把理论焊进项目。

---

## 7. 最短总结

技术伦理学不是“技术人要不要善良”。
它真正问的是：

> **当你的代码开始改变别人的边界、劳动、风险和权力时，你有没有资格按下 Enter？**

对 keikeu 来说，答案很清楚：

**你可以整理自己的饺子醋。
你不能把别人的粮变成你的数据库。**
