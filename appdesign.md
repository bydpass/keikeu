> slogan：存住一瞬的灵光  
> 当前阶段：repo 清空后重启开发  
> 设计原则：本地优先、创作者资产优先、MVP 极简、无 AI 依赖

## 0. 本文用途

本文是 keikeu 新 repo 的应用设计基准文件。

它不负责解释宏大愿景，不负责写商业计划，不负责证明“AI 创作工具有前途”。  
它只回答一件事：

> 第一版 keikeu 到底要做成什么样，什么能进，什么必须滚出去。

旧 repo 已经形成尸山。新开发从本文件重新开局。

---

## 1. 产品定义

keikeu 是一个面向同人创作者的本地优先灵感整理工具。

它的主发心只有一个：

> 当「创作者」记录灵感时，指引记录者将其变为「一份大纲」。

MVP 的核心链路：

```text
乱灵感 / 口嗨
  ↓
灵感 cache
  ↓
配方票填写 / 生成
  ↓
本地 Markdown 大纲
```

一句话定义：

> keikeu 是一个把同人灵感从“口嗨火星子”炼成“可继续写作的大纲文件”的本地工具。

---

## 2. 目标用户

### 2.1 唯一目标用户

keikeu 的目标用户是且仅是：

> 有能力动笔写作的同人创作者，尤其是单机同人女。

这里的「创作者」不是泛内容消费者，不是平台用户，不是约稿甲方，不是社群运营者。

### 2.2 用户特征

目标用户通常具有以下特征：

- 会产生大量片段式灵感、口嗨、场面执念。
- 能写作，但常在“灵感 → 正式动笔”之间卡住。
- 不一定需要复杂项目管理。
- 对隐私、CP、角色、作品资产敏感。
- 可能排斥 AI 直接介入创作。
- 更愿意使用简单、可控、可迁移的工具。
- 需要“把灵感搬出第三方备忘录”的理由。

### 2.3 不是目标用户

MVP 不服务：

- 约稿平台用户
- 接稿工作室
- 同人社区运营者
- AI 代写用户
- 内容推荐消费者
- 需要多人协作的团队
- 需要云同步与账号系统的人

这些人以后也许能用，但不能决定 MVP。

---

## 3. 产品立场

keikeu 的产品立场：

> 创作者资产属于创作者，不属于软件。

因此：

- 第一版必须本地优先。
- 不做账号。
- 不做云同步。
- 不做社区。
- 不接外部 fandom / 角色 / CP 数据库。
- 不默认使用 AI。
- 不把灵感关进不可迁移的黑箱。
- 用户选择本地存储位置。
- cache 与 outline 都保存为 Markdown。
- `keikeu_index.json` 只做辅助索引，不做资产本体。

如果 keikeu 死掉，用户仍然应该能打开自己的 Markdown 文件继续写。

---

## 4. MVP 范围

### 4.1 MVP 只做三件事

1. **记录灵感**
   - 快速写入一段原始灵感 / 口嗨。
   - 保存为本地 Markdown cache 文件。

2. **整理成配方票**
   - 从 cache 创建大纲。
   - 通过 caveman-friendly 表单填写大纲字段。
   - 允许留白，但温和建议补完。

3. **保存为 Markdown 大纲**
   - 用户选择本地存储位置。
   - 输出可读、可复制、可迁移的 `.md` 文件。

### 4.2 MVP 不做什么

MVP 明确不做：

- AI 代写
- 云同步
- 账号系统
- 社区 / 发现流 / 热榜
- 外部 fandom 数据库
- 复杂知识图谱
- 多人协作
- 约稿交易
- 自动发布
- 写作调理
- 灵感浮现 / 灵光回潮
- desktop pet
- 复杂主题系统
- 完整 Obsidian / Notion 替代品

这些功能再香，也先砍。  
MVP 只验证“口嗨 → 大纲”能不能 3 分钟跑通。

## 5. 核心产物：Markdown 大纲

keikeu 的最终产物是一份 Markdown 文件。

### 5.1 大纲字段

大纲包含以下字段：

```markdown
# 标题

## 1. 原始灵感

保留最初的灵感、口嗨、碎片文本。

## 2. 整理后摘要

这篇大概讲什么。

## 3. Fandom + 人物 / CP

- Fandom：
- 人物：
- CP：

## 4. 观前提醒

- 原作 / AU / IF / PA：
- CP 结构：
- 情节元素：

## 5. 流水账

故事大概怎么走。

## 6. Ending Type

HE / BE / OE / 自定义

## 7. 与其他灵感的逻辑关联（Optional）

- 关系：
- 关联对象：
- 说明：
```

### 5.2 字段说明

#### 原始灵感

必须保留。  
这是火种，不允许被整理后摘要覆盖。

#### 整理后摘要

由用户手写。  
未来可以加入 AI 辅助，但 AI 不能成为必要条件。

#### Fandom + 人物 / CP

完全自定义。  
只做本地存储与本地自动补全。  
不得接入外部数据库。

#### 观前提醒

导出 Markdown 时使用“观前提醒”这个同人语境熟词。  
软件内部编辑时可以显示为“内容要素”，避免过早滑向发布语境。

#### 流水账

默认自由文本。  
可以提供模板，但不能强制模板。

MVP 可提供一个简单模板：

```markdown
- 开头：
- 中段：
- 结尾：
```

#### Ending Type

固定枚举：

- HE
- BE
- OE
- 自定义

不加入 TE 等扩展枚举，避免第一版术语污染。

#### 逻辑关联

只允许关联本地已有 cache / outline。  
关系枚举必须少。

MVP 关系枚举：

- 前作
- 续作
- IF
- 外传
- 同系列

不做图谱视图。  
不做复杂节点网络。  
不做“灵感宇宙”。

## 6. 灵感 Cache

### 6.1 Cache 定义

cache 不是完整备忘录系统。  
cache 是灵感的临时收容层，等待被炼成大纲。

> cache 的职责是收住火星子，不是成为第二个 Obsidian。

### 6.2 Cache 文件格式

每条 cache 都保存为 Markdown 文件。

示例：

```markdown
---
type: cache
status: raw
created: 2026-06-19 21:30
updated: 2026-06-19 21:30
linked_outline:
---

# 黄光床边灵感

## 原始灵感

梅蕾尔黄光里醒来，阿德琳站在床边，像等了她很多年。
想写一种“妹妹不是来复仇，是来要被承认”的感觉。

## 临时备注

还没想好 ending。
可能是 OE。
```

### 6.3 Cache 状态

MVP 只保留四种状态：

- `raw`：刚存，未处理
- `drafting`：已开始转配方票
- `outlined`：已生成大纲
- `archived`：封存

不要做复杂标签系统。  
不要做复杂 folder。  
不要做知识库。

### 6.4 Cache 最小字段

- 标题
- 原始灵感
- 临时备注
- 创建时间
- 更新时间
- 状态
- 关联 outline 路径

---

## 7. 本地存储结构

用户首次启动时选择一个本地文件夹作为 keikeu vault。

示例：

```text
keikeu-vault/
  cache/
    2026-06-19-2130-黄光床边灵感.md
  outlines/
    2026-06-19-阿德琳等着姐姐.md
  keikeu_index.json
```

### 7.1 文件夹说明

#### cache/

保存灵感 cache Markdown 文件。

#### outlines/

保存大纲 Markdown 文件。

#### keikeu_index.json

辅助索引文件。  
用于加速列表、搜索、状态筛选、路径关联。

原则：

> Markdown 是资产本体。JSON 是软件索引。

索引坏了，可以从 Markdown 重建。  
Markdown 丢了，索引没有意义。

## 8. 页面设计

MVP 页面缩成两页半。

### 8.1 页面一：灵感 Cache

用途：

> 快速收灵感，并把灵感送入配方票。

核心功能：

- 新建灵感
- 编辑灵感
- 保存灵感
- 转成大纲
- 归档
- 查看状态

列表字段：

- 标题
- 创建时间
- 状态
- 是否已有大纲

禁止事项：

- 不做复杂标签系统
- 不做社交分享
- 不做推荐
- 不做红点催办
- 不做“待处理数量焦虑”

### 8.2 页面二：配方票 Editor

用途：

> 将灵感整理成 Markdown 大纲。

核心字段：

- 原始灵感
- 整理后摘要
- Fandom + 人物 / CP
- 观前提醒 / 内容要素
- 流水账
- Ending Type
- 逻辑关联

核心按钮：

- 保存
- 导出 Markdown
- 从 cache 填入
- 添加逻辑关联
- 检查缺失字段

字段可以留白。  
空字段显示温和提示，例如：

```text
还没想好也行。
```

保存时不阻拦。  
导出时空字段可显示：

```text
待补
```

### 8.3 页面 2.5：本地文件库

不是完整独立模块。  
只是轻量文件列表。

功能：

- 查看 cache
- 查看 outlines
- 搜索标题
- 按状态筛选
- 打开文件
- 在系统文件夹中显示

禁止事项：

- 不做复杂项目管理
- 不做看板
- 不做图谱
- 不做云盘

## 9. 主流程

### 9.1 新建灵感流程

```text
打开 keikeu
  ↓
进入灵感 Cache
  ↓
点击「新建灵感」
  ↓
输入标题 / 原始灵感 / 临时备注
  ↓
保存为 cache Markdown
```

验收：

- 用户能在 30 秒内存下一段灵感。
- 用户能在文件夹里看到对应 Markdown 文件。

### 9.2 灵感转大纲流程

```text
选择一条 cache
  ↓
点击「转成大纲」
  ↓
进入配方票 Editor
  ↓
原始灵感自动填入
  ↓
用户填写其余字段
  ↓
保存为 outline Markdown
  ↓
cache 状态变为 outlined
```

验收：

- 用户能在 3 分钟内从混乱口嗨得到可保存、可编辑的大纲文件。

### 9.3 大纲编辑流程

```text
打开 outline
  ↓
编辑字段
  ↓
保存
  ↓
更新 Markdown 文件与 index
```

验收：

- 用户可以重复打开、编辑、保存。
- 文件始终是普通 Markdown。

## 10. UI 气质

keikeu 的 UI 气质：

- 轻
- 干净
- 不监工
- 不像平台
- 不像企业工具
- 不催办
- 不制造红点焦虑
- 不炫技
- 像一个本地小刀具

按钮文案应直接、温和、无官腔。

推荐文案：

- 新建灵感
- 转成大纲
- 保存
- 导出 Markdown
- 先放着
- 归档
- 检查缺什么

避免文案：

- 立即完成
- 待办
- 任务
- 完成度
- 提升效率
- 智能创作
- AI 赋能
- 一键生成神作

---

## 11. 验收标准

MVP 验收只看一条主任务：

> 用户输入一段混乱灵感，能在 3 分钟内得到一张可保存、可编辑、可继续写作的 Markdown 大纲。

细分验收：

- 首次启动可选择本地存储位置。
- 软件自动创建 vault 结构。
- cache 能保存为 Markdown。
- outline 能保存为 Markdown。
- index 能记录文件状态。
- cache 能转成 outline。
- outline 支持 7 个字段。
- 字段允许留白。
- 软件温和提示补完字段，但不强制。
- 用户能在系统文件夹中找到自己的文件。
- 无账号。
- 无云同步。
- 无外部数据库。
- 无 AI 强依赖。

## 12. Post-MVP / Advance

MVP 完成后，才考虑 advance。

### 12.1 初级 Advance：灵光回潮

功能：

> 用户再次打开软件时，温和浮现旧灵感，询问是否继续整理。

原则：

- 不进入 MVP。
- 不做红点。
- 不做催办。
- 每次最多浮现 1 条。
- 只给三个动作：炼成大纲 / 先放着 / 封存。

### 12.2 初级 Advance：写作调理

功能：

> 帮用户从已有大纲进入写作准备状态。

可能方向：

- 检查大纲缺口
- 提醒冲突不足
- 帮用户拆第一段
- 帮用户整理人物动机
- 帮用户选择开头方式

原则：

- 不代写正文。
- 不压迫用户完成。
- 不变成写作 KPI 系统。

### 12.3 未来可能功能

- 本地模板系统
- 本地词库补全
- 本地全文搜索
- Markdown 预览
- 简单导出包
- 可选 AI 辅助整理
- 可选本地模型接入

全部后置。

## 13. 架构原则

### 13.1 文件优先

Markdown 是主资产。  
JSON 是索引。  
软件围绕用户文件工作，而不是把用户文件吞进软件。

### 13.2 本地优先

所有核心功能离线可用。  
没有网络也能完成主流程。

### 13.3 可迁移

用户可以随时复制整个 vault 文件夹。  
文件可被 Obsidian、VS Code、Typora、普通文本编辑器打开。

### 13.4 低魔法

第一版少做自动判断。  
不猜 fandom。  
不猜 CP。  
不从外部数据库匹配角色。  
不替用户定义作品归属。

### 13.5 小枚举

枚举必须少。  
关系类型少。  
Ending Type 少。  
状态少。

枚举过多会把创作工具变成表格填报系统。

---

## 14. 开发顺序

建议开发顺序：

1. 初始化 vault
2. 创建 cache Markdown
3. 读取 cache 列表
4. 编辑 cache
5. 创建 outline Markdown
6. 配方票 Editor
7. cache 转 outline
8. index.json 写入与读取
9. 本地文件库
10. 缺失字段检查
11. 打开系统文件夹
12. 简单打包

每一步都必须能跑。  
不要先做 UI 皮肤。  
不要先做 advance。  
不要先接 AI。

---

## 15. 当前最高优先级

下一步开发目标：

> 完成一个可以跑通「新建灵感 → 转成大纲 → 保存 Markdown」的最小 demo。

成功标准：

```text
输入：
一段混乱口嗨

输出：
本地 keikeu-vault/outlines/ 下出现一份可读 Markdown 大纲
```

这就是 MVP 的脊椎。

先把脊椎接上，再谈皮肤、肌肉、发光义体。

---

## 16. 总结

keikeu 第一版不是“大而全创作平台”。  
它是一把小刀：

> 收住灵感，切出大纲，把文件交还给创作者。

MVP 的敌人不是功能少。  
MVP 的敌人是贪心、炫技、平台化、AI 依赖、社交污染、文件黑箱。

保持冷酷。  
先让一段口嗨在 3 分钟内变成一份本地 Markdown 大纲。
